# Fix: Command Scheduler Loop Overrun

## Root Cause Analysis

The WPILib command scheduler runs at 50Hz (20ms per cycle). `robotPeriodic()` calls `CommandScheduler.getInstance().run()`, which executes all button polling, command `execute()` methods, and subsystem `periodic()` methods within that 20ms budget. Your code is exceeding this budget.

**Worst-case single cycle (teleop with Y button held):**

| Source | Operation | ~Time |
|--------|-----------|-------|
| FlywheelSubsystem.periodic() | CAN read + 2x SmartDashboard | 3-5ms |
| TurretSubsystem.periodic() | CAN read + SmartDashboard | 2-3ms |
| PivotSubsystem.periodic() | CAN read + SmartDashboard | 2-3ms |
| SwerveSubsystem.periodic() | 4x CAN reads (odometry) | 3-5ms |
| AutoAimCommand.execute() | Limelight NT read + motor cmds | 3-5ms |
| AutoShootCommand.execute() | NT read + CAN read + CAN writes | 3-5ms |
| FeederSubsystem.idleCommand.execute() | 2x CAN writes | 1-2ms |
| Drive command | 4x CAN writes | 2-3ms |
| **Total** | | **~19-31ms** |

Zero headroom. Any CAN contention, NT spike, or GC pause pushes it over.

## Fixes (by priority)

### 1. CRITICAL: Remove `System.out.println` in IntakeSubsystem

**File:** `src/main/java/frc/robot/subsystems/IntakeSubsystem.java` lines 128-131

```java
// REMOVE THIS LINE:
System.out.println("Roller is spinning!");
```

`System.out.println()` on RoboRIO goes through the DS logging subsystem (string formatting + network to DS). Each call takes 1-5ms. At 50 calls/second this alone causes overruns.

### 2. HIGH: Remove or throttle SmartDashboard publishes in periodic()

Three subsystems publish to SmartDashboard every 20ms unconditionally. Each involves CAN read + NT serialization:

- **`FlywheelSubsystem.periodic()`** (lines 76-79) — publishes `Shooter/Target RPM` and `Shooter/Actual RPM`
- **`TurretSubsystem.periodic()`** (line 63) — publishes `Turret/Current Angle`
- **`PivotSubsystem.periodic()`** (line 55) — publishes `Pivot/Current Angle`

**Fix:** Remove these periodic publishes entirely (values are still readable via robot container or command-specific telemetry), OR throttle to every 100ms using a counter:

```java
private int m_periodicCount = 0;
private static final int SMARTDASHBOARD_INTERVAL = 5; // every 5 cycles = 100ms

public void periodic() {
    if (++m_periodicCount >= SMARTDASHBOARD_INTERVAL) {
        m_periodicCount = 0;
        SmartDashboard.putNumber("Shooter/Actual RPM", ...);
    }
}
```

### 3. MEDIUM: Lower YAGSL TelemetryVerbosity

**File:** `src/main/java/frc/robot/subsystems/swervedrive/SwerveSubsystem.java` line 32

```java
// Change from:
SwerveDriveTelemetry.verbosity = TelemetryVerbosity.HIGH;
// To:
SwerveDriveTelemetry.verbosity = TelemetryVerbosity.NONE;
```

`HIGH` publishes extensive swerve data to NT every cycle. On competition, set to `NONE`. Use `LOW` for practice/debugging.

### 4. MEDIUM: Throttle Limelight reads in AutoAim/AutoShoot commands

**Files:**
- `src/main/java/frc/robot/commands/AutoAimCommand.java` line 37
- `src/main/java/frc/robot/commands/AutoShootCommand.java` line 46

`getRawFiducials()` is called every 20ms. The turret/pivot physically cannot move that fast. Cache the data and only re-read every 2-3 cycles (40-60ms):

```java
private int m_execCount = 0;
private RawFiducial[] m_cachedFiducials = new RawFiducial[0];

public void execute() {
    if (++m_execCount >= 3) {
        m_execCount = 0;
        m_cachedFiducials = LimelightHelpers.getRawFiducials(LIMELIGHT_NAME);
    }
    // Use m_cachedFiducials instead of fresh read
}
```

### 5. LOW: Fix `m_feeder.idle()` compilation issue

**File:** `src/main/java/frc/robot/commands/AutoShootCommand.java` lines 52, 88, 97

`FeederSubsystem` has `idleMod()` but no `idle()` method. If this compiles, verify you're running the correct build.

## Verification

After applying fixes:
1. `.\gradlew build` — confirm clean compilation
2. Deploy to robot and run teleop
3. Monitor driver station console for "command scheduler loop overrun" messages
4. If overrun persists, the remaining CAN traffic is the bottleneck — consider configuring slower status frames on non-critical motors (feeder, intake) using Phoenix 6 StatusFrame custom requests
