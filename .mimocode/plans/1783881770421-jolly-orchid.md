# FRC Robot CAN Bus & Loop Overrun Fix (Teleop)

## Problem Summary

During **teleop**, the robot experiences:
1. **CAN bus stale messages** - Devices on "canivore" bus not responding
2. **Loop overrun** - `robotPeriodic()` taking 90ms (4.5x over 20ms budget)
3. **Communication lost** - Driver station loses connection

## Root Cause: CAN Bus Saturation in Teleop

During teleop, every 20ms cycle runs:
- `SwerveSubsystem.periodic()` → Pigeon + 4 encoder reads (5 CAN reads)
- `driveFieldOriented` command → 4 module writes (4 CAN writes)
- `feeder.idleCommand()` → 2 motor writes (2 CAN writes)
- Subsystem periodic methods for Turret, Flywheel, Pivot
- Active button commands (AutoAim, AutoShoot, Intake)

**Total: 12+ CAN transactions per cycle** on a single CANivore bus with 15+ devices.

## Software Fixes

### Fix 1: Throttle Swerve Odometry Updates

**File:** `src/main/java/frc/robot/subsystems/swervedrive/SwerveSubsystem.java`

Reduce `updateOdometry()` from every 20ms to every 40-60ms (2-3x per loop cycle). Full-rate odometry isn't needed for driver control.

```java
private int m_odometryCounter = 0;
private static final int ODOMETRY_INTERVAL = 2; // Update every 2nd cycle (40ms)

@Override
public void periodic() {
    if (++m_odometryCounter >= ODOMETRY_INTERVAL) {
        m_odometryCounter = 0;
        swerveDrive.updateOdometry();
    }
}
```

### Fix 2: Throttle Feeder CAN Writes

**File:** `src/main/java/frc/robot/subsystems/FeederSubsystem.java`

The idle command writes to 2 motors every 20ms. Reduce to every other cycle.

### Fix 3: Batch CAN Signal Refreshes

**File:** `src/main/java/frc/robot/subsystems/swervedrive/SwerveSubsystem.java`

If YAGSL allows, call `BaseStatusSignal.refreshAll()` once per cycle instead of per-device. This batches all status reads into a single CAN transaction.

### Fix 4: Throttle SmartDashboard Writes in Commands

**File:** `src/main/java/frc/robot/commands/AutoAimCommand.java`

Ensure all SmartDashboard calls are throttled (already appears to be every 3rd cycle - verify).
