# Fix: AprilTag Field Layout Orientations

## Problem

All 32 AprilTags in `buildFieldLayout()` use `new Rotation3d()` (zero rotation), assuming every tag faces +X. The official `.fmap` file shows tags have different yaw orientations (0°, ±90°, 180°). This causes PnP to compute wrong robot poses when seeing tags with non-zero yaw — exactly matching the symptom: tag 10 (yaw=0°) works, tag 11 (yaw=-90°) gives wrong pose.

## Root Cause

`Constants.java:123-159` — `buildFieldLayout()` creates all tags with `new Rotation3d()`, ignoring the actual orientations from `FRC2026_ANDYMARK.fmap`.

## Fix

Update every `new AprilTag(...)` in `buildFieldLayout()` to include the correct yaw rotation from the `.fmap` file:

```java
new Rotation3d(0, 0, Math.toRadians(YAW_DEGREES))
```

### File to modify

`src/main/java/frc/robot/Constants.java` — lines 126-157, the `buildFieldLayout()` method.

Each line changes from:
```java
tags.add(new AprilTag(N, new Pose3d(new Translation3d(x, y, z), new Rotation3d())));
```
to:
```java
tags.add(new AprilTag(N, new Pose3d(new Translation3d(x, y, z), new Rotation3d(0, 0, Math.toRadians(YAW)))));
```
