## BUGS (Must Fix)

### 1. Vision P-Gain Hardcoded Instead of Using Constant
**File:** `SwerveSubsystem.java:144`
```java
speeds.omegaRadiansPerSecond = yawError * 0.05; // BUG: hardcoded
```
`Constants.OperatorConstants.AIM_P_GAIN` is defined as `0.05` but never referenced. Should be:
```java
speeds.omegaRadiansPerSecond = yawError * Constants.OperatorConstants.AIM_P_GAIN;
```

### 2. PathPlanner Settings Don't Match Actual Robot Hardware
**File:** `src/main/deploy/pathplanner/settings.json`
- `driveMotorType` says `"NEO"` but robot uses TalonFX (Falcon 500)
- `driveGearing`: `5.143` vs actual `6.75` in physicalproperties.json
- `driveWheelRadius`: `0.048` (≈1.89") vs actual diameter 4" (radius 0.0508m)
- `robotTrackwidth`: `0.546` vs actual 0.4766 (from module locations: 9.383" × 2)
- `maxDriveSpeed`: `5.45` m/s vs `Constants.MAX_SPEED` = 14.5 ft/s ≈ 4.42 m/s

PathPlanner will generate incorrect trajectories if it uses these settings. **Fix all values to match the actual robot.**

### 3. Potential NPE in `Vision.Cameras.getBestResult()`
**File:** `Vision.java:485-489`
```java
double amiguity = bestResult.getBestTarget().getPoseAmbiguity(); // NPE if no targets
```
`getBestTarget()` returns null if `result.getTargets()` is empty. Should check `hasTargets()` first, or use `getTargets().isEmpty()` guard.

### 4. LED Color Channel Order Mismatch
**File:** `LEDSubsystem.java:16,23-27`
```java
m_led = new AddressableLED(0); // Comment says "RBG"
// ...
m_ledBuffer.setRGB(i, r, g, b); // But setRGB expects (index, R, G, B) per WPILib docs
```
If the LEDs are physically wired RBG (as the comment states), using `setRGB` will produce wrong colors. Either:
- Use `setRGB` and remove the "RBG" comment (if wiring is actually RGB), or
- Swap the `g` and `b` parameters in `setRGB` calls, or
- Use `setLED` with explicit R/G/B values in the correct physical order

### 5. `visionDriveTest` Hardcoded, Bypasses `VISION_DRIVE_ENABLED`

### 6. `driveDirectAngle` Declared But Never Used as Default Command Source

## POTENTIAL RUNTIME ISSUES

### 7. `Vision.updateUnreadResults()` — Debounce Time Calculated But Never Used
**File:** `Vision.java:526-543`
```java
double mostRecentTimestamp = resultsList.isEmpty() ? 0.0 : resultsList.get(0).getTimestampSeconds();
double currentTimestamp    = Microseconds.of(NetworkTablesJNI.now()).in(Seconds);
double debounceTime        = Milliseconds.of(15).in(Seconds);
// debounceTime is never used!
```
The debounce logic is incomplete — the method always fetches new results regardless of timing. This may cause unnecessary processing or stale data.

### 8. `Vision.updateEstimatedGlobalPose()` Overwrites Previous Results
**File:** `Vision.java:556-565`
```java
for (var change : resultsList) {
    visionEst = poseEstimator.update(change); // Each iteration OVERWRITES the previous
}
```
Only the last result's pose estimate is kept. If multiple results exist, earlier estimates are discarded. Consider whether only the latest result should be processed, or if the most confident estimate should be kept.

### 9. `SwerveSubsystem.setupPathPlanner()` — Silent Failure
**File:** `SwerveSubsystem.java:95-98`
```java
} catch (Exception e) {
    e.printStackTrace(); // Then continues to schedule warmup command!
}
CommandScheduler.getInstance().schedule(PathfindingCommand.warmupCommand());
```
If `RobotConfig.fromGUISettings()` or `AutoBuilder.configure()` fails, the warmup command is still scheduled. This will cause errors when PathPlanner tries to run. Should return early or throw.

### 10. `AutoBalanceCommand` — No Timeout
**File:** `AutoBalanceCommand.java:57-58`
```java
public boolean isFinished() {
    return controller.atSetpoint();
}
```
If the robot never reaches the setpoint (e.g., stuck on the edge of the charge station), this command runs forever. Add a timeout or a `withTimeout()` wrapper at the call site.

## CONFIG INCONSISTENCY

### 17. `wheelGripCoefficientOfFriction` is Unrealistic
**File:** `physicalproperties.json:14`
```json
"wheelGripCoefficientOfFriction": 1.19
```
FRC rubber wheels on carpet typically have COF ~0.9-1.2, but values above 1.0 can cause the odometry model to predict unrealistic traction. This may cause YAGSL to calculate incorrect swerve kinematics. Consider using a more conservative value like `1.0`.
